## The Problem I Keep Seeing

A few weeks ago I was sitting in the Beacon Theatre watching Tedeschi Trucks Band and fixing bugs in Zabriskie from my phone. I've written about this before — about what it's like to build a social app with Claude as your only collaborator, deploying CSS fixes from your seat while Derek Trucks plays. That workflow is real. I do it all the time. And it works, because the loop is tight: I notice a problem, I describe it, Claude fixes it, I deploy, I verify. One human, one agent, one codebase.

But I kept thinking about what happens when it's not just one agent.

I've been building Zabriskie — a taste-first social network for music, film, and books — by myself with Claude Code for about two months now. 144 commits in a single week. A server-driven UI architecture that lets me push changes without App Store reviews. Live show features, setlist scrapers, audio players, a whole social graph. One developer, one AI collaborator, and a pace that would be impossible without the collaboration.

The thing is, the problems I'm hitting now aren't problems a single agent can solve efficiently. There are bugs that come in from real users while I'm trying to ship new features. There are product decisions that conflict with each other. There are tests that need to be written and infrastructure that needs to be maintained and user feedback that needs to be triaged, all at the same time. The answer, obviously, is more agents. A team. One agent that simulates users and finds friction. One that triages incoming reports. One or more that write code. One that holds the product roadmap and resolves priority conflicts.

The moment you have multiple agents working on the same codebase — reading the same files, writing to the same database, making decisions based on shared state — you have a distributed system. And distributed systems are what I've spent the last decade of my life studying.

I watched two Claude Code instances step on each other's migrations last month. One created migration 267. The other, running in a different worktree, also created migration 267. Different schemas, same filename. The second one silently overwrote the first. This is a lost update — the exact same problem that distributed databases have been solving since the 1970s. It's Lamport's [happened-before relation](https://lamport.azurewebsites.net/pubs/time-clocks.pdf), playing out in a directory of SQL files instead of a network of processes. This is literally the kind of problem [CRDTs](https://hal.inria.fr/inria-00555588) were invented to solve: two independent writers, no coordination, and the system needs to merge their work without losing either update.

That wasn't an isolated incident. I've seen agents make contradictory assumptions about the state of the codebase. I've seen an agent "fix" a bug by reverting a change that another agent made intentionally. I've seen context windows fill up with stale information because no one told the agent that the world had changed since it last looked. These aren't prompt engineering problems. They're coordination problems.

The more I looked at this, the more I realized: nobody is treating multi-agent LLM systems as what they actually are. Distributed systems with unreliable components, partial information, and no shared clock.

## The Moment It Clicked

The migration collision was annoying. But the moment I actually understood what I was looking at came later, while I was reading the [ChatDev paper](https://arxiv.org/abs/2307.07924).

ChatDev is a multi-agent system where LLM agents play different roles — CEO, programmer, art designer, tester — and collaborate through structured dialogues to build software. The role decomposition is smart. The dialogue structure is well-designed. But when I got to the section on how agents share state, I stopped reading.

They pass messages. That's it. There's no persistent shared state beyond the conversation history. If two agents develop contradictory beliefs about the system they're building, nobody detects it until something breaks downstream. There's no causal ordering on the messages — no way to know whether agent A's message was sent before or after agent B's, or whether agent A had seen agent B's earlier message when it sent its own. Imagine ChatDev's "programmer" and "tester" agents working on the same module: the programmer refactors a function signature while the tester is mid-conversation writing assertions against the old one. Neither knows. The tests pass in the tester's context and fail in reality. This is exactly the stale-read problem that distributed databases solved with version vectors decades ago.

And here was ChatDev, [MetaGPT](https://arxiv.org/abs/2308.00352), [AutoGen](https://arxiv.org/abs/2308.08155) — an entire field of multi-agent research — reinventing the same architecture that distributed systems abandoned in the 1990s. Shared mutable state with no concurrency control. Message passing with no causal ordering. No fault model. No formal reasoning about what happens when agents disagree.

I don't say this to diminish their work. These systems made real breakthroughs on the agent layer — role specialization, structured dialogue, tool use patterns, task decomposition. The agents themselves are impressive. But the coordination layer underneath them is where I spent the last decade, and I could see every failure mode I'd ever catalogued hiding just beneath the surface.

That's when I decided this was a research project, not just an observation.

## What I've Spent a Decade Building

In 2015, I was building eventually consistent databases at Basho Technologies. The hard part was never the database engine — it was the merging. How do you take two independent streams of updates and combine them into something consistent without throwing data away?

That question led me to build [Lasp](https://arxiv.org/abs/1510.07191), a programming model on top of Conflict-Free Replicated Data Types — data structures mathematically guaranteed to merge correctly regardless of the order updates arrive or whether the network partitions. No consensus protocol. No leader election. No locking. We ran it on over 1,000 nodes on AWS. It ended up in LoJack's stolen car recovery service across South America. That work received the PPDP 10-year most influential paper award last year.

Then came [Partisan](https://arxiv.org/abs/1802.02652), a distributed runtime that gave us control over the network layer: swap topologies, interpose on every message, inject faults directly into the runtime. And then [Filibuster](https://www.filibuster.cloud), which extracted those fault injection ideas and applied them to microservices — systematically injecting timeouts, connection errors, and unexpected responses into HTTP and gRPC calls to catch bugs during development instead of production.

The thread connecting these projects is the same one I'm pulling on now: how do independent processes coordinate in the presence of partial failure? Lasp answered "make the data structures converge automatically." Partisan and Filibuster answered "verify that convergence actually holds by injecting every failure you can think of." The multi-agent version: how do independent LLM agents share state and communicate without losing work when things go wrong?

## What's Next

I'm calling this project Caucus — independent parties coming together to coordinate. I've spent ten years studying how independent processes coordinate in the presence of failures. The processes used to be microservices, database replicas, Erlang actors. Now they're LLM agents. The problems are the same. The solutions should be too.

Next: [the system design](/ai/agents/distributed/zabriskie/2026/03/31/caucus-system-design.html) — agent roles, the coordination layer, and the experiments I'm running to test whether distributed systems techniques actually improve multi-agent performance.
