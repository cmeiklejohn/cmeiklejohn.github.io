## The Problem I Keep Seeing

A few weeks ago I was sitting in the Beacon Theatre watching Tedeschi Trucks Band and fixing bugs in Zabriskie from my phone. I've [written about this before](/ai/zabriskie/development/2026/03/29/the-show-is-happening-right-now-and-nothing-works.html) — about what it's like to build a social app with Claude as your only collaborator, deploying CSS fixes from your seat while Derek Trucks plays. That workflow is real. I do it all the time. And it works, because the loop is tight: I notice a problem, I describe it, Claude fixes it, I deploy, I verify. One human, one agent, one codebase.

But I kept thinking about what happens when it's not just one agent.

The problems I'm hitting now aren't problems a single agent can solve efficiently. There are bugs that come in from real users while I'm trying to ship new features. There are product decisions that conflict with each other. There are tests that need to be written and infrastructure that needs to be maintained and user feedback that needs to be triaged, all at the same time. The answer, obviously, is more agents. A team. One agent that simulates users and finds friction. One that triages incoming reports. One or more that write code. One that holds the product roadmap and resolves priority conflicts.

The moment you have multiple agents working on the same codebase — reading the same files, writing to the same database, making decisions based on shared state — you have a distributed system. And distributed systems are what I've spent the last decade of my life studying.

I watched two Claude Code instances step on each other's migrations last month. One created migration 267. The other, running in a different worktree, also created migration 267. Different schemas, same filename. The second one silently overwrote the first. This is a lost update — the exact same problem that distributed databases have been solving since the 1970s. It's Lamport's [happened-before relation](https://lamport.azurewebsites.net/pubs/time-clocks.pdf), playing out in a directory of SQL files instead of a network of processes. This is the kind of problem [CRDTs](https://hal.inria.fr/inria-00555588) were invented to solve: two independent writers, no coordination, and the system needs to merge their work without losing either update.

That wasn't an isolated incident. I've seen agents make contradictory assumptions about the state of the codebase. I've seen an agent "fix" a bug by reverting a change that another agent made intentionally. I've seen context windows fill up with stale information because no one told the agent that the world had changed since it last looked. These aren't prompt engineering problems. They're coordination problems.

The more I looked at this, the more I realized: nobody is treating multi-agent LLM systems as what they actually are. Distributed systems with unreliable components, partial information, and no shared clock.

## The Moment It Clicked

The migration collision was annoying. But the moment I actually understood what I was looking at came later, while I was reading the [ChatDev paper](https://arxiv.org/abs/2307.07924).

ChatDev is a multi-agent system where LLM agents play different roles — CEO, CTO, programmer, reviewer, tester, art designer — and collaborate through structured dialogues to build software. The role decomposition is smart. The dialogue structure is well-designed. But when I got to the section on how agents coordinate around shared state, I stopped reading.

ChatDev agents do share artifacts — code, design documents — and it has a mechanism called "communicative dehallucination" where agents reverse roles during code review, with the assistant asking the instructor for clarification before generating code. It's a clever error-reduction heuristic. But it's not concurrency control on those shared artifacts. No causal ordering across chat chains — no way to know whether agent A's modification happened before or after agent B's, or whether agent A had seen agent B's earlier change when it made its own. It's the same problem I saw with my migrations, just wearing different clothes: two agents with stale views of shared state, no mechanism to detect the divergence, and no recovery path when things go wrong. Distributed databases like [Riak](https://docs.riak.com/riak/kv/latest/learn/concepts/causal-context/index.html) and [Antidote](https://www.antidotedb.eu/) solved this with version vectors decades ago.

[MetaGPT](https://arxiv.org/abs/2308.00352) goes further — it introduces a shared message pool where agents publish structured outputs (PRDs, system designs, task lists) and subscribe to relevant messages based on their role profiles. That's a real step forward over ChatDev's dialogue-only coordination. But a publish-subscribe message pool is not concurrency control. It tells you what other agents produced; it doesn't tell you whether you're reading a stale version, or whether two agents are about to write conflicting changes to the same artifact. [AutoGen](https://arxiv.org/abs/2308.08155) stays closer to ChatDev's model — agents coordinate through multi-turn conversations with no persistent shared state at all.

Across this field, the pattern is the same: shared mutable state with no formal concurrency control. No fault model. No reasoning about what happens when agents disagree.

I don't say this to diminish their work. These systems made real breakthroughs on the agent layer — role specialization, structured dialogue, tool use patterns, task decomposition. The agents themselves are impressive. But the coordination layer underneath them is where I spent the last decade, and I could see every failure mode I'd ever catalogued hiding just beneath the surface.

That's when I decided this was a research project, not just an observation.

## What I've Spent a Decade Building

In 2015, I was building eventually consistent databases at Basho Technologies. The hard part was never the database engine — it was the merging. How do you take two independent streams of updates and combine them into something consistent without throwing data away?

That question spawned an entire research program. The foundational work on Conflict-Free Replicated Data Types by [Shapiro et al.](https://hal.inria.fr/inria-00555588) showed that certain data structures are mathematically guaranteed to merge correctly regardless of the order updates arrive or whether the network partitions — no consensus protocol, no leader election, no locking. The EU's [SyncFree project](https://syncfree.lip6.fr) built on that foundation, bringing together researchers across Europe to make CRDTs practical for large-scale systems — producing [Antidote](https://www.antidotedb.eu/), a CRDT-native database, and a body of work on [highly available transactions](https://haslab.wordpress.com/2015/07/07/antidote-the-cure-for-your-cloud-database/) over replicated state.

My contribution was [Lasp](https://arxiv.org/abs/1510.07191), which took the CRDT thinking a step further: instead of just using CRDTs as data structures, Lasp made them the basis for coordination-free distributed programming. Programs in Lasp computed over CRDTs directly — maps, filters, folds — so the entire application inherited their convergence guarantees. We ran it on over 1,000 nodes on AWS. It ended up in LoJack's stolen car recovery service across South America. That work received the PPDP 10-year most influential paper award last year.

Then came [Partisan](https://arxiv.org/abs/1802.02652), a distributed runtime that gave us control over the network layer: swap topologies, interpose on every message, inject faults directly into the runtime. And then [Filibuster](https://www.filibuster.cloud), which extracted those fault injection ideas and applied them to microservices — systematically injecting timeouts, connection errors, and unexpected responses into HTTP and gRPC calls to catch bugs during development instead of production.

The thread connecting these projects is the same one I'm pulling on now: how do independent processes coordinate in the presence of partial failure? Lasp answered "make the data structures converge automatically." Partisan and Filibuster answered "verify that convergence actually holds by injecting every failure you can think of." The multi-agent version: how do independent LLM agents share state and communicate without losing work when things go wrong?

## What's Next

I'm calling this project Caucus — independent parties coming together to coordinate. The processes used to be microservices, database replicas, Erlang actors. Now they're LLM agents. The problems are the same. The solutions should be too.

Next: [the system design](/ai/agents/distributed/zabriskie/2026/04/03/caucus-system-design.html) — agent roles, the coordination layer, and the experiments I'm running to test whether distributed systems techniques actually improve multi-agent performance.
